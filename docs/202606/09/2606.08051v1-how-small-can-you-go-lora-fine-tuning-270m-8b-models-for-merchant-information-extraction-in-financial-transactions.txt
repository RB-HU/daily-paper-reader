Abstract —Financial transaction processing requires extracting structured merchant information from noisy, abbreviated bank transaction strings at scale. Our current production system, a LoRA-fine-tuned LLaMA 3.1-8B, achieves 96.95% F1 on this task, but deploying 8-billion-parameter models imposes prohibitive memory, latency, and cost constraints. To identify more efficient alternatives, we conduct a deployment-focused study of 24 model variants spanning four model families: Gemma 3 (270M, 1B, 4B), Qwen 3.5 (0.8B, 2B, 4B), Aya (3.35B), and LLaMA 3.1-8B— systematically evaluating accuracy, inference throughput, training cost, and hardware behavior to assess pro-duction suitability. Our findings show that: (1) reproducing the LLaMA 3.1-8B fine-tune with a LoRA rank of 8 achieves 96.75% F1, only 0.20 points below the rank-32 baseline; (2) Qwen 3.5 4B with JSON-only prompting reaches 96.60% F1, within 0.35 points of the 8B baseline while using roughly half the parameters; (3) the 0.8B Qwen 3.5 model achieves 94.75% F1, matching models 2.5–4× larger and offering an attractive latency–accuracy trade-off; (4) chain-of-thought fine-tuning generally improves F1 by 0.3-1.8 points across most models, although Qwen 3.5 4B performs best with direct JSON-only prompting; and (5) Qwen 3.5 Think and Nothink training templates produce nearly identical results (F1 differences <0.004), indicating that explicit reasoning super-vision is unnecessary for structured extraction tasks. We further deploy all 14 fine-tuned sub-8B models as Databricks Model Serv-ing endpoints and observe that benchmark performance transfers reliably to production, with an average F1 change of only 0.8 points. Aya 3.35B, based on the Cohere2 architecture, is the sole exception, exhibiting a 3-5 point decline under serving conditions. Based on these results, we provide deployment recommendations across accuracy and latency requirements, demonstrating that compact models can support production financial NLP workloads with approximately half the parameters and up to 4× lower per-sample inference latency. 

Index Terms —Merchant information extraction, financial NLP, LoRA, parameter-efficient fine-tuning, small language models, structured prediction, production deployment 

I. I NTRODUCTION 

Financial transaction processing generates billions of text strings daily, each encoding merchant identity, location, and transaction metadata in compressed, noisy formats. A representative transaction string such as “VIATOR IT-1558003355 360 3RD ST, STE 400, SAN 

*Corresponding author: zxwang@smu.edu.sg 

FRANCISCO 7027495744” must be decomposed into ten distinct entity fields, which include merchant name, location, intermediary, and contact fields (Table I). Accurate extraction is critical for fraud detection, merchant categorization, and consumer analytics [1]. This study positions merchant information extraction as an upstream analytical component for Mastercard products and algorithms that operate on Mastercard-derived transaction descriptor data. The proposed approach is not designed to authorize, route, clear, settle, or otherwise process Mastercard payment transactions; rather, it operates on textual descriptors from existing transactions to improve merchant understanding, data enrichment and analytics, downstream product quality, and the overall reliability of merchant intelligence systems. Our current production system uses a LoRA-fine-tuned LLaMA 3.1-8B [2] that achieves approximately 97% F1 on this task. However, deploying an 8B-parameter model raises practical concerns: high GPU memory requirements (16+ GB in half-precision), limited inference throughput, and substan-tial energy costs [3]. As the system must process millions of transactions daily, even small improvements in efficiency translate to significant operational savings. The proliferation of capable small language models (SLMs)—Gemma 3 [4], Qwen 3.5 [5], Aya [6]—motivates a systematic investigation: what is the minimum model size that maintains production-grade accuracy for structured entity extraction? 

This paper addresses this question from a deployment per-spective. Rather than focusing on methodological novelty, we provide a comprehensive empirical study spanning 24 model variants across four model families, two prompting strategies, and two training templates. Our contributions include: 1) A LoRA rank ablation on LLaMA 3.1-8B showing that rank 8 achieves 96.75% F1—within 0.20 points of the rank-32 baseline—demonstrating that smaller adapters are production-ready. 2) A scaling analysis from 0.27B to 8B parameters, re-vealing that Qwen 3.5 4B closes the accuracy gap to within 0.35 points of the 8B baseline.

II. R ELATED WORK 

A. Parameter-Efficient Fine-Tuning 

LoRA [7] introduced low-rank adaptation as a parameter-efficient alternative to full fine-tuning, and has since been ex-tended by QLoRA [8] for quantized training and AdaLoRA [9] for adaptive rank. In practice these methods are most readily applied through open-source toolkits, and LLaMA-Factory [10] (ACL 2024) is among the most widely adopted: it provides a unified pipeline for parameter-efficient fine-tuning of 100+ open-source LLMs and VLMs, supports a range of training objectives (SFT, DPO, KTO, ORPO, PPO), and typically adds support for new model releases on a day-zero basis. Its widespread adoption has produced a growing body of follow-on research—for example, LoRA fine-tuning of 3.5B–72B open-source LLMs for Chinese-to-English trans-lation [11] and few-shot prompting combined with LoRA fine-tuning of LLMs for logical reasoning on Turtle Soup puzzles [12]—and we adopt it as the fine-tuning backbone of this study. 

B. Small Language Models 

Models such as Phi-3 [13], Gemma 3 [4], and Qwen 3.5 [5] have demonstrated strong performance at 1–4B parameters. Qwen 3.5 introduces a hybrid architecture combining Mamba state-space layers [14] with transformer attention [15]. Con-crete production-oriented evaluations are starting to show that small open-source LMs can match much larger proprietary models on demanding tasks: in LLM-based agentic payment systems, for example, Gemma 4 E4B has been shown to match GPT-5.2’s perfect workflow-fidelity score (ASR) despite being orders of magnitude smaller [16].

A. Task Definition 

Given a raw bank transaction string x, the system produces a JSON object y with ten entity fields (Table I). Each field is either a substring of x or empty. 

B. Dataset 

The dataset contains 8,015 labeled bank transaction strings (8,014 unique), split into 6,125 training, 668 validation, and 1,213 test samples. Entity fields exhibit highly imbalanced presence rates (100% for M ERCHANT _N AME to 5.4% for WEBSITE ). 

C. Models 

We evaluate four model families, all LoRA fine-tuned in this study:  

> •

Gemma 3 [4]: 270M, 1B, 4B (instruction-tuned, standard transformer).  

> •

Qwen 3.5 [5]: 0.8B, 2B, 4B (hybrid Mamba-attention architecture with native extended thinking support).  

> •

Aya [6]: 3.35B ( tiny-aya-global , Cohere2 archi-tecture).  

> •

LLaMA 3.1-8B [2]: 8B (instruction-tuned, standard transformer). Re-fine-tuned with LoRA rank 8 to enable direct comparison against the production baseline. Our production baseline is a separately trained LLaMA 3.1-8B with LoRA rank 32 (F1=96.95%), included as a reference point. We additionally fine-tune LLaMA 3.1-8B with rank 8 under our standardized pipeline so that the 8B model can be compared head-to-head against the smaller candidates in this study. D. Fine-Tuning Pipeline 

All eight models (including our LLaMA 3.1-8B rank-8 fine-tunes) are trained with LLaMA-Factory [10] on an NVIDIA DGX Spark—a compact desktop workstation powered by the GB10 Blackwell Superchip with 128 GB unified LPDDR5x memory. Both training and inference use this single device. This choice deliberately separates experimentation from production. Our Databricks production environment (Sec-tion V) is, by security policy, highly restrictive about which base models may be pulled into the workspace, and its built-in MLflow fine-tuning support lags the open-source ecosystem in covering newly released LLM architectures. LLaMA-Factory, by contrast, is an actively maintained project whose model coverage tracks new releases closely. Running it on a self-contained DGX Spark workstation outside the production perimeter lets us train, evaluate, and select among arbitrary candidate base models without policy or tooling friction. This is also materially cheaper. The 23 retained training runs re-ported here totaled 462 GPU-hours of training plus 195 hours of checkpoint evaluation (12 checkpoints per run) on the DGX Spark—657 GPU-hours of retained compute in all. At the workstation’s manufacturer’s suggested retail price (MSRP), this fixed-cost hardware breaks even against managed cloud GPU compute within a few hundred to a few thousand billable hours depending on instance class and rate; the retained sweep alone approaches that range, and the trial-and-error exploration preceding each retained entry multiplies it further. Only the best-of-family adapters identified through this sweep are subsequently promoted as managed serving endpoints— a two-stage experiment-then-deploy workflow whose viability we confirm via the endpoint validation in Section V. E. Prompt Strategies 

We compare two prompting strategies with identical output labels (full templates in Appendix A): 

Free-Thinking (FT): A two-step chain-of-thought instruction—first analyze freely, then apply entity rules and output JSON. The reasoning is internal only (not in output labels). Prompt: 3,700 chars. 

JSON-Only (JO): Direct extraction with the same rules but no CoT instruction. Prompt: 2,870 chars.

We evaluate each model at all 12 saved checkpoints and report the best. Per-field precision, recall, and F1 are computed via exact string match; the overall F1 is micro-averaged across all entity fields. IV. E XPERIMENTAL RESULTS 

A. Overall Performance 

Table II presents overall performance. The LLaMA 3.1-8B rank-32 baseline achieves 96.95% F1. Our rank-8 LLaMA 3.1-8B fine-tune reaches 96.75% F1 (FT) and 96.57% (JO)— within 0.20 and 0.38 points of the baseline, respectively— demonstrating that rank 8 adapters are sufficient even at 8B. The best sub-8B model is Qwen 3.5 4B with JSON-Only prompting at 96.60% F1—a gap of only 0.35 points from the baseline at half the parameter count. Gemma 3 4B (FT) and both Qwen 3.5 4B Nothink variants (JO NT, FT NT) tie at 96.24%, though the Qwen 3.5 4B variants run 1.7–1.8 × faster. 

B. Scaling Behavior 

Fig. 1 shows F1 scaling with model size. Within Gemma 3 (same architecture at 270M, 1B, 4B), gains are +6.58 points from 270M →1B and +2.31 from 1B →4B, exhibiting dimin-ishing returns above ∼1B. The Qwen 3.5 0.8B model (F1=94.75% with FT No-think) nearly matches the 2B variant (95.18%) and ex-ceeds Gemma 3 1B (93.93%), a model 25% larger with a conventional transformer architecture. We attribute this to Qwen 3.5’s hybrid Mamba-attention design, which combines efficient linear-time sequence modeling with precise positional attention—particularly advantageous for structured extraction where the model must locate specific substrings within noisy input. H. Accuracy–Latency Pareto Analysis 

Fig. 5 plots best F1 against average inference time per sample for all retained variants, revealing the Pareto-optimal configurations. The frontier spans from Gemma 3 270M FT (0.24 s, F1=87.35%) through Gemma 1B FT, Qwen 2B No-think variants, Qwen 3.5 4B JO NT, Qwen 3.5 4B JO (1.97 s, F1=96.60%) to LLaMA 8B FT (7.46 s, F1=96.75%). Many configurations are dominated: Aya 3.35B and Gemma 3 4B achieve lower F1 than similarly-fast Qwen variants, and LLaMA 8B’s slow inference makes it Pareto-optimal only at the accuracy extreme. Most strikingly, switching from LLaMA 8B FT (7.46 s, F1=96.75%) to Qwen 3.5 4B JO (1.97 s, F1=96.60%) at the top of the frontier trades only 0.15 F1 points for a 3.8× reduction in per-sample inference time— the headline efficiency gain motivating sub-8B deployment in production. Benchmark accuracy transfers to production serving. 

Across the 14 endpoints the mean change is only −0.0081 F1. Excluding the two Aya 3.35B variants (discussed below), every endpoint falls within 1.1 points of its benchmark score, and two—Qwen 3.5 2B JO ( +0 .0023 ) and Gemma 3 4B JO (+0 .0007 )—marginally exceed it. This confirms the central premise of the study: the accuracy rankings established un-der local evaluation hold when the models are deployed as managed production endpoints.

Based on our results, we propose tiered deployment strate-gies: 

Tier 1 — Maximum accuracy (>96% F1): for sites willing to keep the 8B footprint, the LLaMA 3.1-8B rank-8 FT fine-tune (96.75% F1) is a production-validated alternative to the rank-32 baseline at 4 × smaller adapter size; endpoint validation of this variant is deferred to future work. For a smaller alternative, Qwen 3.5 4B JO at 96.60% F1 closes to within 0.35 points of the baseline at half the parameter count. Gemma 3 4B FT or Qwen 3.5 4B Nothink variants achieve 96.24% and offer further deployment flexibility. 

Tier 2 — Balanced (>95% F1): Qwen 3.5 2B FT Nothink (95.18% F1, 0.98 s/s) is the recommended choice—strong accuracy with high throughput and endpoint accuracy that matches benchmark ( ∆F1 = −0.0025 in Table IV). Aya 3.35B (95.72% F1, 14.5 h training, 0.33 s/s) edges it on benchmark but loses 3–5 F1 points on the Databricks endpoint due to its Cohere2 architecture (Section V); we therefore advise against Aya for production serving despite its benchmark score. 

Tier 3 — Latency-critical (>94% F1): Qwen 3.5 0.8B FT Nothink achieves 94.75% F1 at 1.32 s/s on DGX Spark— remarkable for a sub-1B model. Its hybrid architecture makes it the clear choice for edge or high-throughput deployments. 

Prompt strategy. Use Free-Thinking for Gemma and Aya models, which provides +0.3–1.8 F1 points with no inference cost increase. For Qwen 3.5 4B, JSON-Only prompting yields the best result (96.60%); for smaller Qwen models, FT and JO perform comparably. 

Training template. For Qwen 3.5, Nothink training is preferred: it matches Think accuracy while offering faster inference (up to 2.9 × for FT prompts) and simpler output parsing (no thinking tags). LoRA configuration. Rank 8 is sufficient across model sizes: rank 32 yields only +0.07 points on Aya 3.35B and +0.20 points on LLaMA 3.1-8B at 4× the adapter size, reducing adapter footprint and simplifying model versioning. 

Production serving. Endpoint validation (Section V) con-firms that benchmark accuracy transfers to Databricks Model Serving for the Gemma and Qwen families (mean ∆F1 

= −0.0081 ). Aya 3.35B is the exception, losing 3–5 F1 points once deployed; we therefore favor the Gemma 3 and Qwen 3.5 families over Aya for production serving, and advise validating any candidate directly on its target endpoint. 

Experiment-then-deploy workflow. Our results support a deliberate two-stage workflow: experiment and select with LLaMA-Factory on flexible compute (DGX Spark in our case), then deploy and serve only the selected adapters via MLflow on a security-constrained production cluster. This separation lets teams evaluate any newly released open-source LLM as soon as its weights become available, while keeping the pro-duction attack surface limited to a small set of vetted models. It also keeps experimentation cost predictable—fixed-hardware amortization rather than per-experiment cloud GPU spend— which matters disproportionately for the hyperparameter trial-and-error inherent in such sweeps. We expect this pattern to generalize to other enterprise settings with similar tooling and policy constraints. VII. C ONCLUSION 

We presented a deployment-oriented study of LoRA-fine-tuned small language models for financial entity extrac-tion, evaluating 24 model variants from 0.27B to 8B pa-rameters across two prompting strategies and two training templates. Reproducing the LLaMA 3.1-8B fine-tune with LoRA rank 8 achieves 96.75% F1—within 0.20 points of the rank-32 baseline—confirming that smaller adapters are production-ready even for the 8B model. The best sub-8B model—Qwen 3.5 4B with JSON-Only prompting—achieves 96.60% F1, closing to within 0.35 points of the 8B baseline. Qwen 3.5’s hybrid Mamba-attention architecture enables com-petitive accuracy even at 0.8B parameters (94.75% F1). Chain-of-thought fine-tuning generally helps across model sizes, though Qwen 3.5 4B demonstrates that sufficiently capable hybrid architectures can match or exceed CoT-trained variants with direct prompting. Think and Nothink training templates produce near-identical accuracy for Qwen 3.5, with Nothink offering faster inference. Deploying the fine-tuned variants as Databricks Model Serving endpoints confirms that these benchmark results transfer to production serving (mean ∆F1 

= −0.0081 ), with the Cohere2-based Aya the sole excep-tion. Together, these findings validate a two-stage experiment-then-deploy workflow—broad experimentation with LLaMA-Factory on flexible compute, followed by production deploy-ment of only the selected adapters via MLflow—that is well suited to enterprise environments where production model availability is constrained by security policy. Overall, the results demonstrate that production financial NLP systems can achieve near-state-of-the-art accuracy with dramatically reduced computational requirements, enabling deployment on consumer-grade hardware. D. JSON Output 

The two prompts diverge in how they describe the output. The FT prompt includes an explicit JSON schema: 

### JSON Output Format Return your Step 2 classification with exactly these keys and nothing else: {"INTERMEDIARY": "", "MERCHANT_NAME": "", "DYNAMIC_DESCRIPTOR": "", "STREET": "", "ZIP": "", "CITY": "", "STATE": "", "COUNTRY": "", "PHONE": "", "WEBSITE": "" }

The JO prompt omits the schema and ends with an instruction only: 

### Output Format Return a JSON object with exactly these keys. Leave any key as an empty string if no value is found. Return only the JSON object with no additional text.
