## 摘要
训练调用API的大型语言模型智能体需要大量高质量轨迹。然而，大规模收集此类数据通常需要具备可执行API和真实预填充后端数据库的完整实现环境，这成为了可扩展性的主要瓶颈。为克服这一限制，我们提出了一种无环境的合成数据生成方法，将大语言模型用作动态的数字世界模型。仅基于API规范，我们的方法就能生成模拟智能体与有状态环境交互的轨迹。具体而言，大语言模型首先生成可利用所提供的API解决的多样化任务。随后，教师智能体逐步解决每个任务，同时一个大语言模型模拟器根据任务上下文和模拟历史生成连贯的合成API响应。最后，由大语言模型评判器对轨迹进行筛选，以确保所得数据集的质量。我们在具有挑战性的AppWorld和OfficeBench基准上评估了该方法，这些基准包含信息检索和状态变更任务。在我们的合成数据上微调模型带来了显著的性能提升，表明可以在没有任何可执行环境的情况下生成有效的调用API智能体所需的监督信号。我们的结果确立了基于大语言模型的API模拟作为一种实用、可扩展的解决方案，用于跨多样的API生态系统训练智能体。

## Abstract
Training API-calling large language model (LLM) agents demands massive amounts of high-quality trajectories. However, collecting such data at scale typically requires fully implemented environments with executable APIs and realistic, pre-populated backend databases, creating a major bottleneck for scalability. To overcome this, we propose an environment-free synthetic data generation approach that leverages LLMs as on-the-fly digital world models. Given only API specifications, our method generates trajectories mimicking interactions between an agent and a stateful environment. Specifically, an LLM first generates diverse tasks solvable with the provided APIs. A teacher agent then iteratively solves each task while an LLM simulator generates coherent synthetic API responses conditioned on the task context and simulation history. Finally, an LLM judge filters the trajectories to ensure the quality of the resulting dataset. We evaluate our approach on the challenging AppWorld and OfficeBench benchmarks, which include both information-retrieval and state-changing tasks. Fine-tuning models on our synthetic data yields significant performance gains, demonstrating that effective supervision for API-calling agents can be generated without any executable environment. Our results establish LLM-based API simulation as a practical, scalable solution for training agents across diverse API ecosystems.

- **根本问题**：能否在**完全不构建底层可执行环境**的情况下，生成高质量智能体训练轨迹？
- **研究动机**：绕过环境构建与数据填充的瓶颈，仅利用 API 规范（名称、用途、输入/输出模式）即可生成模拟交互数据，使数据合成变为一个轻量级“规格问题”，从而无缝扩展到多样 API 生态系统。

### 2. 方法论：ESAT 管线
论文提出**ESAT (Environment-Free Synthetic Agentic Trajectory generation)**，一种无环境合成轨迹生成管线，仅以 API 规范为输入，通过三个阶段产出训练用轨迹。

#### 2.1 阶段一：任务合成 (Task Synthesis)
- **分桶生成 (Bucketized generation)**：以（难度、行动类型、任务焦点、APP 数量、每个 APP 使用 API 数范围）构建组合网格，每桶调用任务生成 LLM 多次，确保覆盖广泛。
- **逆频采样 (Inverse frequency-based sampling)**：维护 APP 和 API 的使用计数，优先采样使用较少的 APP 和 API，避免生成偏向少数高频接口。
- **LLM 评判与重写**：生成任务后由 LLM 评判器过滤无效项；通过的任务再经 LLM 重写为简洁、意图清晰的用户请求（如去掉过程性步骤），并再次评判，保证可解性和自然性。

#### 2.2 阶段二：轨迹合成 (Trajectory Synthesis)
- **教师智能体**：迭代解决任务，生成推理和 API 调用。
- **LLM 模拟器**：扮演数字世界模型，为每个 API 调用动态生成合成响应，不执行任何真实 API。
  - **上下文**：完整 API 规范、调用参数、任务描述、同 APP 之前的交互历史、虚拟用户信息、当前日期时间。
  - **模拟规则**：
    - **数据一致性**：保持跨轮次状态一致（如某歌曲的艺术家不变）。
    - **数据多样性**：列表返回中包含部分任务相关和部分无关项（模拟真实噪声）。
    - **数据真实性**：满足字段约束（数值范围、唯一标识等）。
    - **模式合规**：严格匹配 API 响应模式。
  - **质量检查**：
    - 调用前程序化验证参数（缺失、类型等），失败直接返回错误。
    - 响应 JSON 解析和模式匹配，失败则重试。
    - 对 GET 请求使用 LLM 评判器进行语义一致性检查，反馈修正，限制重试次数。
- **终止条件**：智能体调用完成任务 API，或超过步数上限，或调用非预期 APP 的 API，或模拟失败。

#### 2.3 阶段三：轨迹过滤 (Trajectory Filtering)
- 使用更强的 LLM 评判器（论文中用 Gemini-3.1-Pro）对完整轨迹进行多次评判（如检查逻辑正确性、步骤无冗余、无幻觉），仅保留多数为阳性的轨迹。

### 6. 主要结论与发现
- **无环境合成数据高度有效**：在 AppWorld 和 OfficeBench 上，微调 ESAT 合成数据使模型性能大幅提升，且仅使用合成数据（如 ESAT‑S52）就能超越在真实环境收集的 AWT 数据训练的结果。
- **强泛化能力**：独立于目标 APP 的合成数据可传授鲁棒的 API 调用技能，使较小模型（如 4B/8B）超越 GPT‑4o 等大模型的 zero‑shot 表现，或接近教师模型。
- **轨迹过滤至关重要**：强大的评判器（如 Gemini‑3.1‑Pro）能有效过滤低质量轨迹，而较弱的评判器（如 GLM‑5.1‑FP8）可能降低性能。
- **LLM 模拟器质量高**：根据自动评判，93.7% 的模拟 API 响应符合规范且与历史一致；但长响应（>500 词）时失败率上升。
- **现有简单工具调用数据集**（如 ToolAlpaca、ToolACE、Nemotron）无法有效处理复杂有状态环境，进一步微调真实环境数据后仍落后于 ESAT。

### 7. 优点
- **创新性**：首次提出将 LLM 用作状态一致的“数字世界模型”，在不搭建任何后端的情况下生成复杂多步交互轨迹，将数据合成简化为规格问题。
- **可扩展性**：仅需 API 规范，极大降低了扩展到新 API 领域的工程成本。
- **全面验证**：通过两个高难度基准、多模型规模、与真实环境准确率验证、人类评估以及多种对比方法，充分证明了方法的有效性和合成数据的质量。
- **数据覆盖与平衡**：逆频采样和分桶生成保证了 API 覆盖均匀，避免了数据偏差。
- **详细的消融分析**：对评判器模型影响、模拟失败模式、数据分布、API 覆盖率等进行了详尽分析。

### 8. 不足与局限
- **长响应质量下降**：模拟器在需要生成长文本（>500 词）时失败率升高，可能影响需要返回大量条目的场景，需依赖更强的底层模型。
- **多 APP 轨迹占比失衡**：由于生成器更易产出单应用任务，2‑3 应用任务的轨迹数量显著偏少，可能限制跨应用协作能力的全面学习。
- **模拟器潜在偏差**：LLM 作为模拟器可能带有预训练知识中的偏见，无法完全模拟真实环境的意外错误或边缘情况。
- **计算开销**：合成过程中涉及大量 LLM 调用（生成、评判、模拟器多次重试），推理成本较高，虽然论文强调了可扩展性但未提供具体成本对比。
- **缺乏在线反馈学习**：目前仅使用基于预先合成数据的监督微调，未探索与强化学习等在线方法的结合。
