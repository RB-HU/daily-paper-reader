## 摘要
自进化框架通常优化任务解决方案，而将周围的框架视为固定不变。我们提出面向框架感知的自进化（HASE），这是一种智能体强化学习框架，其中单个模型可以在多轮动作空间中生成任务解决方案或编辑选定的框架组件。HASE 使单个 Qwen3-8B 模型能够在文本分类任务中达到与使用 Claude Code 作为框架提议器的 GPT-OSS-120B 模型相媲美的性能。在 alpha 因子挖掘中，HASE 优于报告的 GPT-OSS-120B 基线。HASE 还能修复不完善的评估组件，并在圆填充算法发现中收敛到最先进的性能。这些结果表明，HASE 通过统一的智能体过程同时改进了框架和解决方案。

## Abstract
Self-evolving frameworks usually optimize task solutions while treating the surrounding harness as fixed. We introduce Harness-Aware Self-Evolving (HASE), an agentic reinforcement-learning framework in which a single model can generate task solutions or edit selected harness components in a multi-turn action space. HASE enables a single Qwen3-8B model to match the text-classification performance of a GPT-OSS-120B model that uses Claude Code as the harness proposer. In alpha factor mining, HASE outperforms the reported GPT-OSS-120B baseline. HASE also repairs imperfect evaluation components and converges to state-of-the-art performance in circle-packing algorithm discovery. These results show that HASE improves the harness and the solution through one unified agentic process.

- **核心问题**：现有的自进化系统（如FunSearch, AlphaEvolve）通常将大语言模型（LLM）与任务之间的“脚手架”或“框架”（Harness）视为固定不变的。该框架包含提示词、记忆体、检索策略、评估器/验证器等关键组件。
- **问题背景与动机**：
    - **固定框架的两大局限**：
        1.  **评估器缺陷**：人工编写的评估器可能不完整或与真实环境目标不一致。在分数驱动优化下，模型会投机取巧（reward hacking），而非解决真实任务。
        2.  **搜索先验不佳**：即使评估器正确，指导性框架（如提示词、记忆体）也可能提供糟糕的搜索引导，例如遗漏关键结构、信息过于冗长或隐藏关键示例。
    - **核心含义**：本文挑战了“固定框架”这一假设，提出框架本身也是可以并且应当被优化的对象。通过让模型同时优化任务解决方案和框架，能够突破性能瓶颈，让更小、更经济的模型达到甚至超越依赖强大外部模型和固定框架系统的性能。

### 2. 论文提出的方法论

- **核心思想**：提出 **HASE (Harness-Aware Self-Evolving)**，一种智能体强化学习框架。其核心是构建一个**统一的动作空间**，让单一模型（策略）既可以生成任务解决方案，也可以编辑框架组件，实现模型权重、框架和任务解决方案的协同进化。
- **关键技术细节**：
    - **两类可编辑框架组件**：
        1.  **指导性组件 (Guidance Components)**：如提示词、记忆格式化、上下文检索等，用于改进搜索本身。
        2.  **评估性组件 (Evaluation Components)**：如验证器和评分器，用于定义任务的解决方案有效性和评分。编辑此类组件需“锚定”真实世界的反馈（Oracle）。
    - **统一动作空间**：模型在每一步可以选择：
        - **提交解决方案**。
        - **使用沙盒工具检查和修改框架**，包括`ls`, `cat`, `grep`, `replace`, `overwrite`等文件操作命令。
    - **算法流程（伪代码描述）**：
        1.  **初始化**：策略模型πθ、初始框架、可选的本地评估器和真实世界评估器。
        2.  **阶段循环**：整个训练分为多个阶段（Phase）。
        3.  **轨迹展开与GRPO更新**：每个阶段包含多个epoch，使用GRPO（群组相对策略优化）更新模型。在一个episode中，模型在统一动作空间中多轮交互，可能临时编辑框架并最终提交解决方案。此时，框架编辑仅在该条轨迹内有效。
        4.  **奖励设计**：
            - 直接对框架操作不给予奖励。
            - 对于指导性组件编辑，采用**概念验证（PoC）评分**：如果一条轨迹编辑框架后，获得的解决方案奖励高于当前固定框架下的基线，则该编辑被视为有前景的候选。
            - 对于评估性组件编辑，仅在本地评估器与真实评估器评分不一致时才给予奖励，奖励大小与解决不一致案例（Mismatch Set）的程度成正比。
        5.  **阶段级审查与框架提交**：在每个阶段结束时，对轨迹中产生的候选框架编辑进行审查，筛选出稳定有效的更新，合并到持久化框架中，供下一阶段所有轨迹使用。评估器补丁只有在修复了Mismatch Set中更多案例时才被接受，实现一种类似迭代调试的修复过程。
