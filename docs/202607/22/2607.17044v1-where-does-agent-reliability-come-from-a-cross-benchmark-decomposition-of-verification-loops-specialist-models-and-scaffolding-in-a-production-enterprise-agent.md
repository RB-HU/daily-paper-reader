## 摘要
多步骤企业智能体任务以特有方式失败：单遍推理在决定答案与提交答案之间没有检查点。我们研究了一个生产系统（Leni），其架构设置了这样的检查点：由轻量级任务特化后训练模型组成的验证循环（执行、观察、比较、纠正）。我们在三个公共基准上评估了未经修改的生产配置，这些基准分别侧重不同的失败模式：SpreadsheetBench Verified（静默计算错误），BullshitBench v2（前提编造）和GAIA验证集（长工具链中的级联错误）。与前沿基座模型相比，完整系统在SpreadsheetBench上提升了11.0个百分点（91.25%对80.25%，n=400，p<0.001），在BullshitBench上提升了7至10个百分点（98%对91%，n=100），在GAIA验证集上提升了约15个百分点（75.2% pass@1，n=165；83.0% best-of-k）。我们的核心贡献是对该提升进行了分解：大部分提升来自脚手架、路由和专家模型，而非验证步骤本身，后者的独立贡献很小（+1.5个百分点），但集中在分数分布的顶端，能够挽救原本失败的任务。我们对循环进行了端到端的测量，得到了一个实证的验证器混淆矩阵（捕获率约0.20，修复率0.75，无误报退化），该矩阵为复合可靠性模型奠定了基础。专家替换消融实验表明，循环的价值取决于谁在观察：用生成式前沿模型替换小型训练后的验证器将消除大部分挽回效果。一个有效前提控制实验显示，在100个专家级别的问题中，零过度拒绝。

## Abstract
Multi-step enterprise agent tasks fail in a characteristic way: single-pass inference has no checkpoint between deciding an answer and committing to it. We study one production system (Leni) whose architecture installs such checkpoints: verification loops (execute, observe, compare, correct) staffed by lightweight task-specialized post-trained models. We evaluate the unmodified production configuration on three public benchmarks stressing distinct failure modes: SpreadsheetBench Verified (silent computation error), BullshitBench v2 (premise confabulation), and the GAIA validation split (cascade error over long tool chains). The full system improves over its frontier base model by +11.0 percentage points on SpreadsheetBench (91.25% vs 80.25%, n=400, p<0.001), +7 to +10 percentage points on BullshitBench (98% vs 91%, n=100), and roughly +15 points on GAIA validation (75.2% pass@1, n=165; 83.0% best-of-k). Our central contribution is a decomposition of that uplift: most of it comes from scaffolding, routing, and specialist models rather than from the verification step itself, whose isolated contribution is small (+1.5 points) but concentrated at the top of the score distribution, where it converts otherwise-failing tasks. We instrument the loop end-to-end, yielding an empirical verifier confusion matrix (catch rate about 0.20, fix rate 0.75, no false-alarm regressions) that grounds a compounding-reliability model. Specialist-swap ablations suggest that the loop's value depends on who observes it: replacing the small trained verifier with the generating frontier model eliminates most rescues. A valid-premise control shows zero over-rejections in 100 expert-level questions.

- **核心问题**：企业级多步智能体（agent）任务因缺少中间检查点，在长链推理或工具调用时容易出现静默错误、前提编造和级联失败。单纯等待更强的基座模型无法系统性地解决此类可靠性问题。
- **整体含义**：论文探索通过**系统架构层面的验证循环**（verification loop）、脚手架、轻量级任务专用模型组合，弥补基座模型的可靠性短板，并试图回答“智能体的可靠性增益究竟来自架构的哪个部分”。

## 2. 论文提出的方法论

- **核心思想**：为智能体执行流程插入“执行→独立观察→与意图比较→纠正”的结构化循环，将不可见缺陷转变为可检测偏差，且观察与比较阶段交由小型、独立后训练的专业模型完成，而非复用生成模型本身。
- **验证循环的四阶段定义**：
  1. **Execute**：基座模型执行动作（如编写电子表格）。
  2. **Observe**：通过独立代码路径重新读取结果（如用 LibreOffice 重新计算公式、提取值）。
  3. **Compare**：将观测结果与任务意图比对，由轻量级专业模型判断是否存在差异。
  4. **Correct**：发现差异时定向修复并再次观察，直到一致后提交。
- **可靠性模型**：将循环抽象为端到端可靠性 `Rloop = (p')^n`，其中有效单步成功率 `p' = p(1 − f b) + (1 − p) c r`，引入了捕获率 `c`、修复率 `r`、误警率 `f`、误警破坏率 `b`，从而将验证器性能量化为可测量的混淆矩阵。
- **验证神谕的三个实例**：
  - **确定性神谕**（电子表格重新计算）：用 LibreOffice 重算公式作为外部真值，配合 Leni-Cell-S 专家模型进行单元格比对。
  - **自反思神谕**（认知防火墙）：将问题分解为前提申明，由 Leni-Triage-S 分类为已知有效/看似合理/误用，拒绝编造前提。
  - **规划器中介神谕**（GAIA 规划回路）：规划器与执行器分离，执行器输出带类型 artifact（值、引用、错误类），规划器比对后重新规划或纠正。
- **轻量专家模型栈**：基于 Qwen3（0.5B–4B）后训练得到 Cell-S、Triage-S、Parse-S、Route-XS，使用蒸馏 SFT + RLVR/GRPO，量化至4比特，成本仅前沿模型的 0.02–0.1 倍。

## 6. 论文的主要结论与发现

- **架构提升显著但来源分明**：总提升可达+7~+15 pp，但分解显示大部分来自**脚手架和路由**，验证循环的独立贡献仅约+1.5 pp（SpreadsheetBench）。然而这点微小增益集中在高分端，能将原本失败的最困难任务挽救回来。
- **观察者的独立性决定循环价值**：将小型专用验证器换成生成 artifact 的模型本身时，解救数大幅下降（6→2），说明效果来自独立、专门训练的小模型，而非单纯的“循环”结构。这印证了模型自评估偏见的已有发现。
- **验证器混淆矩阵**：从生产循环中实测得到捕获率 `c≈0.20`、修复率 `r≈0.75`、无观测误报 (`f≈0`)，证明该循环保守且精准，不会破坏正确结果。
- **成本可行性**：小模型使“每一步都验证”成为可负担的默认操作，而非昂贵附加项。
- **链长预测未证实**：模型预测长链中循环收益更大，但本次实验未能在 GAIA 中明确验证，留作可证伪假说。
